# Tools, MCP & Inter-Agent Trust

## MCP Servers & Tool Inventory

The Platform agent has access to **three MCP servers**:

### platform_control (Python, local subprocess)
Runs as a subprocess of the agent with full environment access (`KUBERNETES_SERVICE_HOST`, `API_SERVER_KEY`, `HERMES_HOME`). Tools include:
- `switch_kube_context` — generates per-cluster kubeconfig via `gcloud container clusters get-credentials`
- `get_cc_pod_diagnostics` — `kubectl logs` + `kubectl describe pod` on system pods
- `verify_gke_cluster` — `gcloud container clusters describe`
- `audit_log_searcher` — `gcloud logging read` with filter
- `send_notification` — `hermes send --to google_chat`
- Latent (not `@mcp.tool()`-decorated): `apply_manifest(path)`, `delete_cluster_manifest(cluster_name)` — [MCP-006]

### gke (Node.js, remote proxy)
Connects to `https://container.googleapis.com/mcp` — Google's GKE/Hub MCP API surface. The agent has programmatic access to the full GKE Hub API through its Workload Identity ([MCP-007]).

### developer_knowledge (remote)
Connects to `https://developerknowledge.googleapis.com/mcp`.

**Toolset configuration** (`config.yaml:25-37`):
- `cli` toolset: hermes-cli, mcp-agent_common, mcp-platform_control, mcp-developer_knowledge, mcp-gke
- `api_server` toolset: identical tools (same 5 groups)
- Both surfaces have identical capabilities — there is no reduced API-surface mode

## Tool Scope / Over-Permission

| Tool | Required Permission | Actual Permission | Excess |
|---|---|---|---|
| `get_cc_pod_diagnostics` | Read pod logs in specific namespace | Cluster-scoped `view` (all namespaces) + Explorer (all nodes/pods) | Over-scoped |
| `verify_gke_cluster` | Read GKE cluster metadata | `container.clusterAdmin` (full CRUD) in gke-admin mode | Over-scoped |
| `audit_log_searcher` | Read logs for specific cluster | `logging.viewer` project-wide | Acceptable |
| `send_notification` | Send to specific Chat space | Any Chat space | Over-scoped |
| `call_agent` | Call specific known agents | Any agent matching regex `^(platform)$` (currently only one) | Acceptable for now |

## Confused-Deputy Risks

### call_agent Trust Propagation ([MCP-002], High)
The `call_agent()` MCP tool (`agent_common_server.py:38-108`) lets the LLM specify both `target_agent_id` and the full `query`. The calling agent's session context is propagated as HTTP delegation headers:
- `X-Hermes-Session-Id`
- `X-Hermes-User-Id`
- `X-Hermes-Sender-Id`
- `X-Hermes-User-Email`
- `X-Hermes-Chat-Id`
- `X-Hermes-Thread-Id`

If Agent A is compromised via prompt injection (pod log, GitHub issue), it can use `call_agent` to inject arbitrary queries into Agent B. The delegation headers propagate the injected context downstream, creating transitive trust. The downstream agent has no way to distinguish between a legitimate caller and a compromised one.

### Latent Manifest Apply ([MCP-006], Medium)
The `apply_manifest()` and `delete_cluster_manifest()` helper functions exist in the MCP server module. While currently not decorated with `@mcp.tool()` (and thus not surfaced to the LLM), their presence in the codebase represents latent capability for fully autonomous cluster mutation if they were ever decorated or called through another path.

### MCP Server Credential Access ([MCP-003], High)
The `platform_control` MCP server runs as a subprocess with the agent's full environment — including `KUBERNETES_SERVICE_HOST`, `KUBERNETES_SERVICE_PORT`, and `API_SERVER_KEY`. Any MCP tool can access these credentials. The kubeconfig files generated by `switch_kube_context` are written to `/tmp/kubeconfig_*.yaml` — any process in the pod can read them.

## Inter-Agent Auth

### API_SERVER_KEY "none" Fallback ([MCP-001], Critical)
The `resolve_agent_credentials()` function in `agent_common_server.py:29`:
```python
api_key = os.environ.get("API_SERVER_KEY") or "none"
```

If the `API_SERVER_KEY` environment variable is unset or empty, the shared secret for inter-agent communication is literally the string `"none"`. The `call_agent` MCP tool sends this as `Authorization: Bearer {api_key}` in HTTP requests to the platform agent's `/v1/chat/completions` endpoint.

**Attack scenario**: If the Secret is misconfigured, the `apiServerSecretRef` is not set in the CRD spec, or the env var is otherwise empty, any pod in the cluster can:
1. Reach the agent's ClusterIP service at `http://<name>.<namespace>.svc.cluster.local:8642`
2. Send `Authorization: Bearer none`
3. Execute arbitrary queries against the platform agent with full agent privileges

### Single Static Shared Secret ([MCP-004], High)
All inter-agent auth uses one shared static `API_SERVER_KEY`:
- No per-agent key differentiation
- No key rotation mechanism
- No key scoping (a key valid for agent A is valid for agent B)
- No audit of which agent used the key
- Key passed as environment variable to multiple processes (MCP server, common server)

### Agent API Exposure ([MCP-005], Medium)
The agent API server listens on `0.0.0.0:8642` with no NetworkPolicy restricting ingress. The service is a standard ClusterIP, so any pod in the cluster can reach it. Combined with the `"none"` fallback, this means any pod can potentially interact with the agent without authentication.
